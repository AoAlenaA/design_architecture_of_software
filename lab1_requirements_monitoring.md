# Лабораторная работа №1  
**Тема:** Формулирование требований к программной системе  

**Вариант / предметная область:**  
«Система мониторинга и аналитики эффективности адаптации менеджеров по продажам в ИИ-платформе обучения персонала»

**Цель работы:**  
Научиться анализировать поставленную задачу, формулировать функциональные и нефункциональные требования к модулю аналитики и отчетности в составе ИИ-платформы адаптации менеджеров по продажам.

## 2. Перечень функциональных требований

Ниже — функциональные требования к модулю мониторинга и аналитики (далее — «Модуль»). Для удобства каждому требованию присвоен идентификатор FR-N.

### 2.1 Требования со стороны руководителя отдела продаж

1. **FR-1. Дашборд «Состояние адаптации»**  
   Модуль должен предоставлять РОПу дашборд с текущим состоянием адаптации всех стажёров: статус (в процессе / завершено / не завершено), процент прохождения программы, текущий рейтинг, дата последней активности.

2. **FR-2. Детальная карточка стажёра**  
   Модуль должен позволять РОПу открывать карточку конкретного стажёра с отображением:
   - истории прохождения тестов (дата, результат, количество попыток);
   - статистики по диалоговому тренажёру (количество сценариев, частота критических ошибок, типичные возражения, с которыми стажёр не справляется);
   - динамики ключевых метрик адаптации во времени.

3. **FR-3. Аналитика по группе / команде**  
   Модуль должен формировать отчёты по группе стажёров (например, «набор августа 2025»), включая:
   - средний прогресс адаптации по группе;
   - распределение по уровням знаний;
   - сравнение групп между собой по выбранным периодам.

4. **FR-4. Отчёты по этапам воронки и знаниям продукта**  
   Модуль должен предоставлять отчёты:
   - по этапам воронки продаж (приветствие, выявление потребностей, презентация, обработка возражений, закрытие);
   - по знаниям продукта (категории продукта, тарифы, ограничения и т.п.);
   с отображением типичных ошибок и «узких мест» по каждому этапу.

5. **FR-5. «Тепловая карта ошибок»**  
   Модуль должен отображать «тепловую карту» ошибок, где по осям представлены:
   - темы / этапы (знание продукта, этапы воронки, типы возражений, соблюдение скрипта),
   - стажёры или группы,  
   а цвет/интенсивность отражает частоту ошибок.

6. **FR-6. Настройка фильтров и срезов**  
   Модуль должен позволять РОПу фильтровать и агрегировать данные по:
   - периоду времени;
   - группе/набору;
   - конкретному стажёру;
   - типу метрики (знание продукта, дисциплина по скрипту, возражения и т.п.).

7. **FR-7. Экспорт отчётов**  
   Модуль должен предоставлять возможность экспорта основных отчётов в форматах, пригодных для отчётности (например, PDF и CSV/Excel).

---

### 2.2 Требования со стороны HR-специалиста

8. **FR-8. Отчёты по завершённости адаптации**  
   Модуль должен формировать отчёты для HR по степени завершения адаптации: кто прошёл программу, кто в процессе, кто «завис» (нет активности больше N дней).

9. **FR-9. Отчёт по качеству набора**  
   Модуль должен предоставлять отчёт, позволяющий оценивать качество набора по источникам найма (если данные доступны): доля успешно завершивших адаптацию стажёров по каждому источнику.

10. **FR-10. Отчёт для руководства**  
    Модуль должен формировать агрегированный отчёт «для директора» в виде:
    - ключевых метрик адаптации (среднее время до выхода на план, доля успешно прошедших адаптацию);
    - сравнения периодов «до внедрения системы» и «после внедрения».

---

### 2.3 Требования к интеграции и данным

11. **FR-11. Импорт результатов тестов**  
    Модуль должен автоматически получать данные о прохождении тестов из модуля генерации тестов (Студент 1) через описанный API или обмен сообщениями (например, по расписанию или в режиме near-real-time).

12. **FR-12. Импорт данных диалогового тренажёра**  
    Модуль должен получать агрегированные данные по результатам диалогов (успешность сценариев, количество критических ошибок, типы возражений) из модуля диалогового тренажёра (Студент 2).

13. **FR-13. Хранение истории активности**  
    Модуль должен хранить историю активности стажёра (прохождения тестов, сценариев тренажёра, изменения метрик) за период не менее N месяцев (конкретное значение будет уточнено).

14. **FR-14. Модель данных для метрик**  
    Модуль должен иметь модель данных, позволяющую:
    - добавлять новые типы метрик без серьёзного изменения схемы БД;
    - связывать метрики с конкретными стажёрами, группами, периодами и источниками данных.

---

### 2.4 Требования по управлению доступом и конфигурации

15. **FR-15. Ролевая модель доступа**  
    Модуль должен поддерживать разделение ролей: РОП, HR, топ-менеджмент, администратор, с разными правами просмотра и управления.

16. **FR-16. Настройка метрик и отчетов**  
    Модуль должен позволять администратору и/или РОПу настраивать:
    - набор отображаемых метрик;
    - пороговые значения для выделения «проблемных» стажёров;
    - расписание автоматического формирования и рассылки отчётов (например, еженедельный отчёт по отделу).

17. **FR-17. Журналирование действий пользователей**  
    Модуль должен вести журнал действий (кто и когда просматривал/изменял настройки отчётов, экспортировал данные и т.п.) для целей аудита.

## 4. Перечень сделанных предположений

Здесь фиксируются допущения, которые **не были явно оговорены** в постановке задачи, но влияют на формирование требований (особенно нефункциональных):

1. **Объём пользователей и данных**
   - Одновременно системой пользуются до 10–30 руководителей и HR-специалистов.
   - Общее количество стажёров, по которым хранится история, — до 500–2000 человек.
   - Объём данных по тестам и диалогам растёт, но не превышает десятки миллионов записей в год.

2. **Режим работы системы**
   - Система используется в основном в рабочее время (например, 9:00–21:00 по будням).
   - Допускается кратковременное снижение доступности ночью для регламентных работ.

3. **Интеграции с другими модулями**
   - Модули генерации тестов и диалоговый тренажёр уже реализуют API для передачи результатов в модуль аналитики.
   - Формат данных между модулями согласован и может быть расширен без полного пересмотра интеграции.

4. **Требования по безопасности и данным**
   - Система обрабатывает персональные данные сотрудников (ФИО, должность), поэтому должна соответствовать действующему законодательству по защите ПДн (например, ФЗ-152 и внутренним политикам безопасности компании).
   - Доступ к системе осуществляется только по защищённому протоколу (HTTPS).

5. **Техническая платформа**
   - Модуль является частью корпоративного веб-приложения (браузерный интерфейс).
   - Пользователи работают из корпоративной сети или через VPN.

6. **«До» и «после» внедрения системы**
   - Предполагается наличие исторических данных по адаптации «до системы» (даже если они частично собраны вручную), либо будут определены метрики и подход к их приблизительной оценке.

Эти предположения далее используются при выборе и конкретизации нефункциональных требований.

## 5. Перечень нефункциональных требований

В соответствии с подходами из литературы по архитектуре ПО (Ричардс, Форд «Фундаментальный подход к программной архитектуре» — рассматриваются архитектурные свойства вроде производительности, отказоустойчивости, безопасности и др.) выделим 5 **наиболее значимых и приоритетных** для данного модуля нефункциональных требований.

### 5.1 Производительность и отзывчивость

**NFR-1. Время отклика основных сценариев**

- Дашборд адаптации (UC-1) и отчёт по стажёру (UC-2) должны загружаться:
  - не более чем за **3 секунды** при типовой нагрузке (до 30 одновременных пользователей);
  - не более чем за **5 секунд** при пиковой нагрузке (до 50 одновременных пользователей).
- Формирование сложных отчётов (по группам, «тепловых карт») допускает большее время построения (до 10 секунд), при этом пользователю должно отображаться явное состояние загрузки (индикатор).

**Обоснование:** руководителю важно быстро получать информацию для оперативных решений; долгие отчёты снижают ценность системы.

---

### 5.2 Масштабируемость и расширяемость

**NFR-2. Возможность добавления новых метрик без пересборки системы**

- Модель данных и архитектура модуля должны поддерживать добавление новых метрик (например, новые типы ошибок или этапы воронки) без полной переработки схемы БД и пользовательского интерфейса.
- Добавление новой метрики должно сводиться к конфигурации (описанию: источник данных, тип, способ отображения) либо к локальному расширению, не влияющему на существующие сценарии.

**Обоснование:** по мере развития отдела продаж появляются новые сценарии, продукты и подходы, и система аналитики должна «подстраиваться» под них.

---

### 5.3 Надёжность и доступность

**NFR-3. Доступность и целостность данных**

- Целевые показатели доступности модуля в рабочее время — не ниже **99 %** в месяц.
- В случае сбоев система должна:
  - корректно обрабатывать ошибки внешних модулей (недоступность API), логировать их и повторять попытки импорта данных;
  - обеспечивать отсутствие потерь уже сохранённых данных (за счёт резервного копирования БД не реже одного раза в сутки, либо средствами используемой БД/платформы).

**Обоснование:** потеря данных по адаптации и исторических метрик делает систему недостоверной и снижает доверие пользователей.

---

### 5.4 Безопасность и конфиденциальность

**NFR-4. Защита персональных данных и управление доступом**

- Все соединения с модулем должны выполняться по протоколу HTTPS.
- Доступ к аналитике должен быть защищён авторизацией и ролями:
  - РОП видит данные только по своему отделу;
  - HR может видеть данные по всем сотрудникам, но без «лишних» технических деталей;
  - директор видит агрегированные отчёты по компании;
  - администратор может управлять пользователями и ролями, но не обязательно видеть все персональные данные.
- Действия пользователей (авторизация, экспорт данных, изменение настроек) должны журналироваться.

**Обоснование:** статистика адаптации привязана к конкретным сотрудникам, её утечка потенциально нарушает закон и внутренние политики безопасности.

---

### 5.5 Удобство использования (юзабилити)

**NFR-5. Понятность и наглядность интерфейса**

- Основные дашборды и отчёты должны быть доступны пользователю **не более чем в 2–3 клика** от главной страницы.
- Интерфейс должен содержать:
  - наглядные визуализации (диаграммы, графики, тепловые карты);
  - возможные подсказки/легенду по цветам, метрикам и статусам;
  - сохранение пользовательских фильтров (период, группа, тип метрик) между сессиями.
- Интерфейс должен быть адаптирован под типичное рабочее разрешение экрана (например, 1920×1080).

**Обоснование:** РОП и HR — не аналитики-программисты; слишком сложные и «загромождённые» отчёты будут редко использоваться, а значит — модуль не принесёт ожидаемую пользу.
